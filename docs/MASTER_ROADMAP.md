# NEXTEP ERP — Master Product Roadmap

> - **الحالة:** المرجع الرسمي لتطوير المنتج
> - **الإصدار:** 1.0
> - **المالك:** Product Management & Engineering Lead

هذا المستند هو المصدر الرسمي للحقيقة (**Single Source of Truth**) لخارطة طريق NEXTEP ERP. لا يبدأ Sprint جديد، ولا تدخل Feature إلى التنفيذ، إلا إذا كانت مسجلة هنا أو في الـBacklog المرتبط به، ومحددة الأولوية والنطاق ومعايير القبول والاعتماد.

## Vision

### ما هو NEXTEP ERP؟

NEXTEP ERP هو نظام عربي متكامل لإدارة المصنع والمشاريع والعمليات التشغيلية والمالية والموارد البشرية من مساحة عمل واحدة آمنة، لحظية، وقابلة للتوسع. صُمم ليكون نظام التشغيل اليومي للمصنع، وليس مجرد مجموعة شاشات منفصلة لتسجيل البيانات.

### ما المشكلة التي يحلها؟

يستبدل النظام تشتت البيانات بين الملفات الورقية وجداول البيانات وتطبيقات المحادثة والأنظمة المنفصلة بمصدر موحد وموثوق. يهدف إلى معالجة:

- صعوبة معرفة الحالة الفعلية للمشاريع والإنتاج والمخزون والالتزامات المالية.
- تكرار الإدخال وتعارض البيانات بين الإدارات.
- غياب المساءلة عن التعديلات والاعتمادات والعمليات الحساسة.
- بطء انتقال المعلومة بين الإدارة والمصنع والمالية والموارد البشرية.
- الاعتماد على إجراءات يدوية يصعب قياسها أو توسيعها.
- ضعف تجربة الاستخدام في الأنظمة غير المصممة للعربية وبيئات العمل اليومية الطويلة.

### من هم المستخدمون؟

- **المالك والإدارة العليا:** متابعة الأداء والمخاطر والمؤشرات واتخاذ القرار.
- **مديرو المشاريع:** إدارة دورة المشروع والمهام والتسليم والتكلفة والملفات.
- **الإنتاج والتشغيل:** تنفيذ أوامر الإنتاج ومتابعة المواد والمعدات وحالة العمل.
- **المالية والمحاسبة:** المشتريات والمصروفات والمبيعات والمستحقات والرواتب والتقارير.
- **الموارد البشرية:** الموظفون والعمالة اليومية والحضور والورديات والاستحقاقات.
- **المخازن والمشتريات:** الأرصدة والحركات والتوريد والاستلام والمسؤوليات.
- **الموظفون الميدانيون:** تنفيذ الإجراءات المسموح بها بسرعة من الحاسب أو الهاتف.
- **المراجعون ومسؤولو النظام:** إدارة الوصول ومراجعة سجل التدقيق والامتثال.

### أهداف النظام

1. إنشاء مصدر بيانات واحد ودقيق لكل عمليات المصنع.
2. تقليل العمل اليدوي والأخطاء ووقت الوصول إلى المعلومة.
3. جعل حالة العمل والتغييرات مرئية لحظيًا للمستخدمين المصرح لهم.
4. تطبيق صلاحيات دقيقة، وفصل مسؤوليات، وسجل تدقيق كامل.
5. دعم دورة العمل من التخطيط إلى التنفيذ والاعتماد والإغلاق.
6. تقديم تجربة عربية RTL واضحة ومريحة للاستخدام اليومي الطويل.
7. توفير أساس Modular وقابل للتوسع للشركات والفروع والتكاملات المستقبلية.
8. تحويل البيانات التشغيلية إلى مؤشرات تساعد الإدارة على اتخاذ قرارات قابلة للقياس.

## Product Principles

### Security First

الأمان جزء من التصميم منذ التحليل، وليس مراجعة نهائية فقط. أقل صلاحية ممكنة هي الوضع الافتراضي، وتُطبق الحماية في قاعدة البيانات والـAPI والواجهة معًا.

### Simplicity

المستخدم يصل إلى الوظيفة الأساسية خلال خطوتين أو ثلاث قدر الإمكان. نقلل الحقول والخيارات والقرارات غير الضرورية، ونُظهر المعلومات الأهم أولًا.

### Realtime by Default

أي بيانات تشغيلية مشتركة أو صلاحيات مؤثرة يجب أن تتزامن بين الجلسات دون Refresh أو إعادة تسجيل الدخول، مع معالجة واضحة للانقطاع وإعادة الاتصال.

### Audit Everything

كل عملية حساسة من إضافة أو تعديل أو حذف أو اعتماد أو تغيير صلاحية يجب أن تكون قابلة للتتبع بهوية المنفذ والتوقيت والكيان والقيم المتأثرة.

### Scalable Architecture

تُبنى القرارات لتدعم زيادة المستخدمين والبيانات والوحدات دون إعادة كتابة النظام. نفصل المسؤوليات، ونمنع التكرار، ونقيس الأداء قبل أن يصبح عائقًا.

### Modular ERP

كل وحدة لها حدود واضحة وعقد بيانات وصلاحيات مستقلة، مع تكامل مدروس مع بقية الوحدات. لا تُنشأ تبعيات خفية بين الشاشات.

### Mobile Ready

كل تدفق جديد يُصمم من البداية للاستجابة للمقاسات المختلفة، واللمس، والاتصال غير المستقر، حتى قبل إصدار تطبيق الهاتف الأصلي.

### Arabic First + RTL

العربية وRTL هما تجربة أساسية وليستا ترجمة لاحقة. المصطلحات موحدة، والمحاذاة والتسلسل البصري والجداول والنماذج تُراجع في RTL.

### Performance First

يُقاس أثر كل Feature على حجم الحزمة، وعدد الاستعلامات، وزمن التحميل، وإعادة الرسم. نستخدم التحميل الكسول والتخزين المؤقت والفهارس عندما يثبت الاحتياج.

## Development Rules

كل Feature جديدة تمر بالمراحل التالية بالترتيب، ولا يتم تخطي أي مرحلة:

1. **Analysis** — المشكلة، المستخدم، القيمة، النطاق، المخاطر، الاعتماديات، ومعايير النجاح.
2. **UX/UI Design** — رحلة المستخدم، الحالات الأساسية والاستثنائية، التصميم المتجاوب، RTL، وإمكانية الوصول.
3. **Database Design** — نموذج البيانات، القيود، الفهارس، migrations، الملكية، وسياسة الاحتفاظ.
4. **Backend** — الخدمات والاستعلامات والعقود والتحقق ومعالجة الأخطاء وعمليات Realtime.
5. **Frontend** — الواجهة والحالة والتغذية الراجعة والصلاحيات والأداء والاتساق مع Design System.
6. **Security Review** — RLS، المصادقة، التفويض، فصل الصلاحيات، التخزين، الأسرار، وسيناريوهات إساءة الاستخدام.
7. **Testing** — Unit، Integration، سيناريوهات الصلاحيات، حالات الفشل، الاختبار المتجاوب، والقبول متعدد المستخدمين عند الحاجة.
8. **Documentation** — توثيق المستخدم والتشغيل والـmigrations والقرارات المعمارية وتحديث هذه الخارطة عند تغير النطاق.
9. **Pull Request** — PR محدود النطاق بوصف واضح، وخطة اختبار، وأثر قاعدة البيانات، وخطة الرجوع.
10. **Merge** — بعد نجاح الفحوصات، اكتمال المراجعة، إغلاق الملاحظات، واعتماد صاحب القرار المناسب.

### بوابات الانتقال

| المرحلة | المخرج الإلزامي قبل الانتقال |
| --- | --- |
| Analysis | Product Brief ومعايير قبول قابلة للاختبار |
| UX/UI Design | تدفق معتمد يشمل Loading وEmpty وError وPermission states |
| Database Design | مخطط وMigration قابلة للرجوع أو خطة استعادة موثقة |
| Backend | عقد واضح، تحقق من المدخلات، وأخطاء قابلة للتشخيص |
| Frontend | تطبيق متجاوب RTL لا يتجاوز صلاحيات المستخدم |
| Security Review | Checklist أمان وRLS واختبارات وصول ناجحة |
| Testing | الأدلة والنتائج وكل الاختبارات المطلوبة ناجحة |
| Documentation | توثيق التشغيل والتغيير والمخاطر محدث |
| Pull Request | مراجعة بشرية وفحوصات CI ناجحة |
| Merge | موافقة نهائية وعدم وجود ملاحظات مانعة |

### قواعد إضافية ملزمة

- لا يُربط بدء التنفيذ بتاريخ قبل اعتماد Analysis وUX/UI Design.
- لا يُعدل production schema يدويًا؛ كل تغيير عبر migration مرتبة وقابلة للمراجعة.
- لا تعتمد الواجهة وحدها في حماية البيانات؛ التفويض يُطبق في Supabase/RLS أيضًا.
- لا تستخدم `window.location.reload()` لمزامنة الحالة.
- لا تُدمج Feature ناقصة خلف واجهة ظاهرة؛ إن لزم، تستخدم Feature Flag موثقة.
- أي تغيير واسع النطاق يُقسم إلى PRs صغيرة قابلة للمراجعة دون كسر التوافق الخلفي.
- كل Feature لها Product Owner وEngineering Owner ومعيار نجاح بعد الإطلاق.

## Backlog

أي فكرة جديدة لا يتم تنفيذها مباشرة. تُسجل أولًا في الـBacklog، ثم تُراجع وتُرتب في جلسة Product Planning قبل إدخالها إلى Milestone أو Sprint.

### الحقول الإلزامية

- **Priority:** `P0 Critical`، `P1 High`، `P2 Medium`، `P3 Low`.
- **Business Value:** القيمة المتوقعة والمستخدم أو العملية المستفيدة، بدرجة من 1 إلى 5.
- **Technical Complexity:** `S`، `M`، `L`، `XL` مع توضيح المخاطر والاعتماديات.
- **Status:** `Idea`، `Discovery`، `Ready`، `Planned`، `In Progress`، `Blocked`، `Done`، `Rejected`.

### قواعد ترتيب الـBacklog

1. تُعطى الأولوية للأمان، وفقد البيانات، وتعطل العمليات، والامتثال قبل التحسينات الشكلية.
2. تُقارن القيمة بتكلفة التنفيذ والمخاطر، ولا تعتمد الأولوية على عدد الطلبات فقط.
3. لا تنتقل البطاقة إلى `Ready` دون مشكلة واضحة ومعايير قبول واعتماديات معروفة.
4. لا تنتقل إلى `In Progress` دون مساحة معتمدة داخل Sprint ومالك واضح.
5. تُراجع العناصر القديمة دوريًا، وتُرفض أو تُؤرشف الأفكار التي فقدت قيمتها.

## Definition of Ready

قبل بدء تنفيذ أي Feature يجب أن:

- تكون المشكلة والمستخدم والقيمة محددة.
- يكون النطاق وما هو خارج النطاق واضحين.
- تكون معايير القبول قابلة للاختبار.
- تكون UX/UI والحالات الاستثنائية معتمدة.
- تكون متطلبات البيانات والصلاحيات وRealtime معروفة.
- تكون الاعتماديات والمخاطر وخطة الإطلاق محددة.
- يوافق Product Owner وEngineering Owner على البدء.

## Definition of Done

لا تعتبر أي Feature منتهية إلا إذا:

- الكود مكتمل ومتوافق مع المعمارية وDesign System.
- الاختبارات المطلوبة ناجحة، بما فيها حالات الفشل والصلاحيات.
- Build ناجح دون أخطاء مانعة.
- Security Review مكتملة وملاحظاتها مغلقة.
- Documentation محدثة للمستخدم والتشغيل وقاعدة البيانات عند الحاجة.
- PR مفتوح بوصف وخطة اختبار وأثر واضحين.
- تمت المراجعة وإغلاق كل الملاحظات المانعة.
- تم الدمج في الفرع الأساسي.
- تم التحقق من نجاح النشر أو migration في البيئة المستهدفة عند انطباق ذلك.
- تم قياس معيار النجاح أو تحديد موعد وطريقة قياسه بعد الإطلاق.

## Roadmap Governance

- **Product Management** يملك الأولويات والنتائج والنطاق ومعايير النجاح.
- **Engineering Lead** يملك الجودة المعمارية والأمان والتقدير وخطة التنفيذ.
- **Design** يملك رحلة المستخدم والاتساق وإمكانية الوصول وRTL.
- أي تغيير في Milestone أو ترتيب Phase أو نطاق معتمد يجب أن يحدث عبر PR يحدّث هذا الملف ويشرح السبب والأثر.
- تتم مراجعة الخارطة في بداية كل Sprint وعند نهاية كل Milestone، مع تحديث الحالة والتواريخ والمخاطر.
- عند التعارض بين وثيقة أخرى وهذه الخارطة في الأولويات أو عملية التطوير، تكون هذه الخارطة هي المرجع، ويُفتح PR لتسوية التعارض.

## Sprint 2 Entry Gate

يُحظر بدء Sprint 2 أو تنفيذ أي Feature جديدة قبل:

1. مراجعة هذا المستند.
2. اعتماد ودمج PR الخاص به في `main`.
3. تحويل نطاق Phase 1 إلى عناصر Backlog مكتملة الحقول.
4. اعتماد Definition of Ready لكل عنصر مختار للـSprint.
